# 多用户权限设计

本文档描述 WebChat 从单管理员客服系统改造成多用户分权限系统后的目标行为。本文只定义产品规则、权限边界、命令交互和关键数据约束，不包含具体实现步骤。

## 目标

实现多用户自助使用客服系统：

- 用户可以管理自己的 `key`、客服群、客户机器人、离线提示、欢迎文案等配置。
- 普通用户、VIP 用户、管理员拥有不同功能权限。
- 管理员可以查询、修改、禁用用户，并管理用户名下的 `key`。
- 客户侧机器人 `/start` 支持按 `key` 显示自定义欢迎文案和使用帮助链接。
- 系统继续保持现有网页客服、Telegram 客服群话题、来源统计、快捷回复和动态客户机器人轮询能力。

## 角色和权限

| 角色 | key 数量 | 网页客服 | 统计 | 自动回复 | 管理范围 |
| --- | ---: | --- | --- | --- | --- |
| 普通用户 | 1 | 不可用 | 不可用 | 不可用 | 只管理自己的 `key` |
| VIP 用户 | 5 | 可用 | 可用 | 可用 | 只管理自己的 `key` |
| 管理员 | 不限制 | 可用 | 可用 | 可用 | 管理所有用户和 `key` |

说明：

- 普通用户可以创建 `key`，也可以绑定客服群和客户机器人，但网页入口不可用。
- 普通用户的 `key` 不能通过 `/<key>` 和 `/widget/<key>` 对外提供网页客服。
- 普通用户不能查看统计，也不能配置快捷回复。
- VIP 用户可以使用网页客服、统计和快捷回复。
- 管理员拥有所有 VIP 功能，并额外拥有用户管理和全局配置能力。

## 多租户边界

`key` 仍然保持全局唯一，因为网页入口路径是 `/<key>`。两个用户不能创建同名 `key`。

所有用户命令都必须校验所有权：

- 当前 Telegram 用户是否存在且启用。
- 当前 Telegram 用户是否拥有该 `key`。
- 当前用户角色是否允许使用该功能。

管理员命令可以绕过所有权限制，但仍需显式使用管理员命令，避免普通管理命令误操作其他用户数据。

## 数据模型

### `users`

新增用户表。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `telegram_user_id` | INTEGER PRIMARY KEY | Telegram 用户 ID。 |
| `username` | TEXT | Telegram 用户名，可为空。 |
| `display_name` | TEXT | Telegram 显示名。 |
| `role` | TEXT | `normal`、`vip` 或 `admin`。 |
| `enabled` | INTEGER | 是否启用，`1` 表示启用。 |
| `vip_until` | TEXT | VIP 到期时间，空值表示不限制或非 VIP。 |
| `created_at` | TEXT | 创建时间。 |
| `updated_at` | TEXT | 更新时间。 |

管理员初始化：

- `config.ADMIN_IDS` 中的用户启动后自动视为管理员。
- 如果数据库里已有同一用户记录，可以将其角色同步为 `admin`，避免管理员失去权限。

### `widgets`

现有表新增字段。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `owner_user_id` | INTEGER | 所属用户的 Telegram 用户 ID。 |
| `welcome_text` | TEXT | 客户侧机器人 `/start` 欢迎文案。 |

建议不要新增独立的 `web_enabled` 字段，网页客服是否可用优先由用户角色决定：

- `normal`：网页客服不可用。
- `vip` / `admin`：网页客服可用。

这样普通用户升级为 VIP 后，已有 `key` 可以立即启用网页客服，不需要额外迁移。

### `bot_bindings`

现有表新增字段。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `owner_user_id` | INTEGER | 所属用户的 Telegram 用户 ID。 |

约束：

- `bot_token` 仍然全局唯一。
- 同一个客户机器人 token 只能绑定到一个 `key`。
- 重新绑定同一个 token 时，应更新到新的 `key` 和新的所属用户。

### `system_settings`

新增全局配置表。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `key` | TEXT PRIMARY KEY | 配置名。 |
| `value` | TEXT | 配置值。 |
| `updated_at` | TEXT | 更新时间。 |

建议配置项：

- `help_link`：客户侧机器人 `/start` 中显示的使用帮助链接。
- `admin_contact`：普通用户和 VIP 用户 `/help` 中显示的管理员联系方式。

### `pending_actions`

新增临时流程表，用于等待用户输入下一条消息。

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `telegram_user_id` | INTEGER | 发起流程的 Telegram 用户 ID。 |
| `action` | TEXT | 当前动作，例如 `await_token`、`await_welcome`、`await_group_key`。 |
| `key` | TEXT | 关联的 `key`，可为空。 |
| `payload` | TEXT | JSON 扩展信息。 |
| `expires_at` | TEXT | 过期时间。 |
| `created_at` | TEXT | 创建时间。 |

使用场景：

- `/tokenadd <key>` 后等待用户发送客户机器人 token。
- `/welcome <key>` 后等待用户发送欢迎文案。
- 群绑定按钮跳转私聊后，等待用户输入要绑定的 `key`。

## 命令设计

### 基础命令

| 命令 | 权限 | 说明 |
| --- | --- | --- |
| `/start` | 所有人 | 根据身份显示入口信息、帮助链接和管理员入口提示。 |
| `/help` | 所有人 | 根据身份显示可用命令和管理员联系方式。 |
| `/myinfo` | 注册用户 | 查看自己的账号、角色、key 数量和所有 key 概览，不包含统计数据。 |
| `/id` | 所有人 | 在群里返回 `chat_id`、群类型、是否论坛、话题 ID。 |

`/start` 行为：

- 普通用户：显示普通功能、VIP 不可用能力和管理员联系方式。
- VIP 用户：显示 VIP 可用功能。
- 管理员：显示普通入口，并提示可使用 `/adminhelp` 查看管理员命令。

### 用户 key 管理命令

| 命令 | 权限 | 说明 |
| --- | --- | --- |
| `/keyadd <key> <显示名>` | 注册用户 | 创建自己的 `key`。普通用户最多 1 个，VIP 最多 5 个，管理员不限。 |
| `/keyinfo <key>` | key 所有者 / 管理员 | 查看某个 `key` 的配置，不包含统计。 |
| `/keydel <key>` | key 所有者 / 管理员 | 删除自己的 `key` 及相关绑定。 |
| `/kon <key>` | key 所有者 / 管理员 | 将 `key` 设置为在线。 |
| `/koff <key> [离线提示]` | key 所有者 / 管理员 | 将 `key` 设置为离线，可同时设置离线提示。 |
| `/kmsg <key> <离线提示>` | key 所有者 / 管理员 | 更新离线提示，不改变在线状态。 |
| `/welcome <key>` | key 所有者 / 管理员 | 进入欢迎文案设置流程，等待下一条消息作为欢迎文案。 |

`/kls` 调整：

- 不再支持不带参数列出所有 `key`。
- 建议废弃 `/kls`，改用 `/myinfo` 和 `/keyinfo <key>`。
- 如果保留 `/kls`，必须要求 `/kls <key>`，否则提示使用 `/myinfo`。

### 客户机器人绑定命令

| 命令 | 权限 | 说明 |
| --- | --- | --- |
| `/tokenadd <key>` | key 所有者 / 管理员 | 进入 token 绑定流程，等待用户发送客户机器人 token。 |
| `/botdel <key>` | key 所有者 / 管理员 | 删除该 `key` 的客户机器人绑定，并停止运行中的客户机器人轮询。 |
| `/botls <key>` | key 所有者 / 管理员 | 查看该 `key` 的客户机器人绑定。 |

`/tokenadd <key>` 流程：

1. 用户向后台管理机器人发送 `/tokenadd yaoyao`。
2. 系统校验 `key` 是否属于当前用户。
3. 系统创建 `pending_actions` 记录，动作是 `await_token`。
4. 系统提示用户在 5 分钟内发送客户机器人 token。
5. 用户发送 token。
6. 系统调用 Telegram `getMe` 验证 token。
7. 系统保存 `bot_bindings`，立即注册到内存并启动客户机器人轮询。
8. 系统尝试删除用户发送 token 的消息，降低泄露风险。
9. 系统回复绑定结果。

### 客服群绑定命令

| 命令 | 权限 | 说明 |
| --- | --- | --- |
| `/groupbind <key>` | key 所有者 / 管理员 | 在客服群中执行，将当前群绑定到指定 `key`。 |

推荐体验：

1. 用户把后台管理机器人拉进客服群。
2. 机器人检测到进群事件后，在群里提示需要绑定 `key`。
3. 群消息中提供「绑定 key」按钮。
4. 用户点击按钮跳转到后台管理机器人私聊。
5. 系统提示用户输入要绑定的 `key`。
6. 用户输入 `key` 后，系统校验所有权并写入当前群 ID。

第一版也可以先支持手动命令：

```text
/groupbind yaoyao
```

该命令只能在超级群中使用，系统直接读取当前 `chat.id`，避免用户手动填写错误的群 ID。

### VIP 功能命令

| 命令 | 权限 | 说明 |
| --- | --- | --- |
| `/qradd <key> <标题>|<答案>` | VIP / 管理员 | 添加快捷回复。 |
| `/qrls <key>` | VIP / 管理员 | 查看快捷回复。 |
| `/qrdel <key> <编号>` | VIP / 管理员 | 删除快捷回复。 |
| `/stats <key> [来源]` | VIP / 管理员 | 查看来源统计。 |
| `/statdel <key> [来源]` | VIP / 管理员 | 清理统计。 |

普通用户执行这些命令时，返回 VIP 提示和管理员联系方式。

### 管理员命令

管理员命令集中放在 `/adminhelp` 中显示。

| 命令 | 说明 |
| --- | --- |
| `/adminhelp` | 显示全部管理员命令。 |
| `/userls [normal|vip|admin|disabled]` | 查询用户列表。 |
| `/userget <telegram_user_id>` | 查询单个用户信息。 |
| `/userset <telegram_user_id> <normal|vip|admin>` | 修改用户角色。 |
| `/userban <telegram_user_id>` | 禁用用户。 |
| `/userunban <telegram_user_id>` | 启用用户。 |
| `/userkeys <telegram_user_id>` | 查看某个用户的所有 `key`。 |
| `/adminkeyinfo <key>` | 查看任意 `key` 的完整信息。 |
| `/adminkeydel <key>` | 删除任意 `key`。 |
| `/helplink <URL>` | 修改全局使用帮助链接。 |
| `/admincontact <联系方式文本>` | 修改管理员联系方式。 |

管理员手动给用户开通 VIP：

```text
/userset 123456789 vip
```

支付功能不在本期范围内。

## 客户侧机器人 `/start`

客户访问某个绑定到 `key` 的客户机器人并发送 `/start` 时：

1. 系统根据客户机器人 token 找到绑定的 `key`。
2. 系统读取该 `key` 的 `welcome_text`。
3. 系统读取全局 `help_link`。
4. 系统回复欢迎文案和使用帮助链接。
5. 如果 `key` 所属用户是 VIP 或管理员，并且该 `key` 配置了快捷回复，则显示快捷回复按钮。
6. 如果用户发送 `/start link123`，`link123` 仍然作为来源参数记录点击。

示例：

```text
欢迎咨询客服瑶瑶。

使用帮助：https://example.com/help
请选择常见问题，或直接发送消息联系人工客服。
```

普通用户的客户机器人不显示快捷回复按钮。

## 网页客服权限

网页入口包括：

- `GET /<key>`
- `GET /widget/<key>`
- `POST /api/msg/<key>`
- `GET /api/history/<key>`
- `GET /api/stream/<session_id>`

权限规则：

- `key` 不存在：返回 `KEY_NOT_FOUND`。
- `key` 存在但所属用户是普通用户：网页客服不可用。
- `key` 所属用户被禁用：网页客服不可用。
- `key` 所属用户是 VIP 或管理员：网页客服可用。

建议对普通用户的网页入口返回 `403 WEB_DISABLED`，方便排查；如果更重视隐藏信息，也可以返回 `404 KEY_NOT_FOUND`。

## 自动退群策略

第一版不建议自动退群。

原因：

- 用户拉机器人进群后可能还没完成绑定，自动退群容易误伤。
- 非客服群消息可以直接忽略，运行成本较低。
- 自动退群策略需要处理群权限、按钮流程、误拉群等边界，容易扩大改造范围。

建议第一版策略：

- 后台管理机器人进群后提示绑定 `key`。
- 未绑定的群中，机器人只响应 `/id` 和 `/groupbind <key>`，其他消息忽略。
- 后续如果垃圾群明显增多，再新增管理员命令手动清理或自动退群策略。

## 安全要求

- 用户发送客户机器人 token 后，系统应尽量删除该消息。
- token 不应出现在 `/myinfo`、`/keyinfo`、`/botls` 的完整输出中，只显示机器人用户名或脱敏 token。
- 所有命令都必须校验用户是否启用。
- 所有 `key` 相关命令都必须校验所有权或管理员权限。
- 所有网页 API 都必须校验 `key` 所属用户的角色和启用状态。
- 管理员命令必须只允许管理员执行。
- 普通用户升级为 VIP 后，应能立即使用此前创建的 `key` 的网页客服、统计和快捷回复能力。

## 非目标

本期不做以下内容：

- 支付功能。
- 网页后台。
- 自动 VIP 到期扣费。
- 多客服坐席账号。
- 更复杂的套餐体系。
- 自动退群强策略。
